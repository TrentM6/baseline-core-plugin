## Document Structure

(Your conventions for structuring documents. Example:)
(- One H1 per document — clear and specific, no clever wordplay)
(- H2 for sections, H3 for subsections, don't go deeper than H3)
(- Every document starts with a one-paragraph summary or TL;DR)
(- End with clear next steps or a call to action)

## Header Conventions

(How you write and use headers. Example:)
(- Headers should be descriptive enough to understand the document from headers alone)
(- Use sentence case, not title case — "How it works" not "How It Works")
(- No punctuation in headers)
(- Numbered headers for sequential processes, unnumbered for reference)

## Paragraphs & Text

(Rules for body text. Example:)
(- 2-3 sentences per paragraph max)
(- No walls of text — break up long sections)
(- Everything should be scannable from headers alone)
(- Lead with the conclusion, then provide supporting details)
(- One idea per paragraph)

## Lists & Tables

(Rules for structured content. Example:)
(- One line per list item, parallel structure)
(- No more than 7 items without grouping into sub-lists)
(- Bulleted lists for unordered items, numbered for sequences)
(- Tables: 2-4 columns, short cells, clear headers)
(- Tables for comparisons and structured data; lists for everything else)
(- No nested lists deeper than 2 levels)

## Emphasis & Notation

(How you use formatting to highlight content. Example:)
(- **Bold** for key terms, labels, and important callouts)
(- *Italic* for quotes, titles, and slight emphasis)
(- `Code` for technical terms, file names, commands, and variable names)
(- Blockquotes for key insights, takeaways, and customer quotes)
(- ~~Strikethrough~~ only for tracking changes, not for style)
(- Use `---` horizontal rules between major sections)

## Links & References

(How you handle links and cross-references. Example:)
(- Inline links with descriptive text — "see the [pricing page]" not "click [here]")
(- External links noted with domain when helpful — "per [Gartner's 2024 report]")
(- Internal cross-references use relative paths)
(- Footnotes for citations in long-form content)

## Code & Technical Content

(If your documents include code or technical content. Example:)
(- Code blocks with language tags for syntax highlighting)
(- Inline code for single terms: `variable`, `function()`, `file.md`)
(- Command-line examples prefixed with `$` for shell commands)
(- Output examples clearly labeled)

## Document Types

(Define the structure for each type of document you commonly produce. Example:)

### PRD / Product Requirements
(- Problem Statement → Context → Requirements → Success Metrics → Open Questions)
(- Requirements table: ID, description, priority, status)
(- Always include "What we're NOT building" section)

### Status Update / Stakeholder Update
(- TL;DR (3 bullets max) → Progress → Blockers → Next Steps)
(- Include dates and owners for action items)
(- Link to relevant documents, don't paste content)

### Research Report
(- Summary → Methodology → Key Findings → Detailed Findings → Recommendations)
(- Findings-first: lead with what you learned, not how you learned it)
(- Include raw data or quotes in appendix)

### Meeting Notes
(- Date, attendees, purpose → Decisions Made → Action Items → Discussion Notes)
(- Action items: who, what, by when)
(- Keep brief — capture decisions and actions, not conversation)

### Proposal / Brief
(- Context → Recommendation → Supporting Evidence → Next Steps)
(- Lead with what you're recommending and why)
(- One page if possible, appendix for detail)

### Blog Post / Article
(- Hook (first paragraph) → Main argument → Supporting points → Conclusion/CTA)
(- Short paragraphs, subheadings every 2-3 paragraphs)
(- Conversational tone per voice.md)

### Email
(- Subject line: specific and actionable)
(- First line: why you're writing)
(- Body: context and detail)
(- Last line: clear ask or next step)
(- Keep under 5 paragraphs)

## RACI Matrix Template

### How to Use

- **R** = Responsible (does the work)
- **A** = Accountable (owns the decision — exactly one per row)
- **C** = Consulted (provides input before decision)
- **I** = Informed (told after the decision)

### Matrix

| Deliverable/Decision | [Role 1] | [Role 2] | [Role 3] | [Role 4] |
|---------------------|----------|----------|----------|----------|
| [Deliverable 1] | R | A | C | I |
| [Deliverable 2] | C | R | A | I |
| [Decision 1] | I | C | R | A |

### Common Mistakes

- Multiple "A" in one row — someone has to own it
- Missing "R" — who's actually doing the work?
- Everyone is "C" — too many cooks, make some "I"
- No "I" row — stakeholders surprised by decisions

## Risk Register Template

### Risk Identification

| ID | Risk | Category | Likelihood | Impact | Score | Mitigation | Owner | Status |
|----|------|----------|-----------|--------|-------|-----------|-------|--------|
| R1 | [What could go wrong] | [Technical/People/External/Scope] | [1-5] | [1-5] | [L×I] | [What we'll do about it] | [Name] | [Open/Mitigating/Closed] |
| R2 | | | | | | | | |

### Scoring Guide

**Likelihood:**
- 1 = Rare (< 10%)
- 2 = Unlikely (10-30%)
- 3 = Possible (30-60%)
- 4 = Likely (60-85%)
- 5 = Almost certain (> 85%)

**Impact:**
- 1 = Negligible (minor delay, easy workaround)
- 2 = Minor (delays one workstream, recoverable)
- 3 = Moderate (delays project, needs escalation)
- 4 = Major (significant scope/timeline impact)
- 5 = Critical (project failure, strategic impact)

**Response by score:**
- 1-5: Accept — monitor and move on
- 6-12: Mitigate — have a plan, review regularly
- 13-19: Actively manage — assign owner, track weekly
- 20-25: Escalate — leadership involvement, consider stopping

## Project Kickoff Checklist

### Before Kickoff

- [ ] Project charter or brief written and approved
- [ ] Stakeholders identified and RACI drafted
- [ ] High-level scope defined (in scope / out of scope)
- [ ] Success criteria documented
- [ ] Initial timeline drafted
- [ ] Team assembled and capacity confirmed
- [ ] Communication plan outlined
- [ ] Kickoff agenda prepared and sent

### Kickoff Meeting Agenda (60-90 min)

1. **Why** — Business context, problem, opportunity (10 min)
2. **What** — Scope, deliverables, success metrics (15 min)
3. **Who** — Team, roles, RACI, stakeholders (10 min)
4. **How** — Approach, methodology, tools (15 min)
5. **When** — Timeline, milestones, key dates (10 min)
6. **Risks** — Known risks, assumptions, dependencies (10 min)
7. **Q&A** — Open questions, parking lot (10 min)
8. **Next steps** — First actions, next meeting (5 min)

### After Kickoff

- [ ] Meeting notes distributed within 24 hours
- [ ] Action items assigned with owners and dates
- [ ] Project workspace set up (tools, channels, docs)
- [ ] Recurring meetings scheduled
- [ ] Risk register created
- [ ] First sprint or phase planned

## Definition of Done Checklist

### Task/Story Level

- [ ] Acceptance criteria met (all items verified)
- [ ] Code reviewed (if applicable)
- [ ] Tested against edge cases
- [ ] Documentation updated
- [ ] No known defects
- [ ] Stakeholder approval (if required)

### Sprint Level

- [ ] All committed stories meet task-level DoD
- [ ] Sprint goal achieved
- [ ] Demo completed
- [ ] Sprint review documented
- [ ] Retrospective completed
- [ ] Action items from retro assigned

### Release Level

- [ ] All sprint-level DoD met
- [ ] Integration testing passed
- [ ] Performance acceptable
- [ ] Security review complete (if applicable)
- [ ] Release notes written
- [ ] Stakeholders notified
- [ ] Rollback plan documented

### Adapting DoD

Every team should define their own DoD. Start with the checklist above, then:
1. Add items specific to your domain (regulatory, compliance, accessibility)
2. Remove items that don't apply
3. Review and update DoD quarterly
4. If items are consistently skipped, discuss whether they belong
